1. What is Bug Bounty?

Bug bounty is a reward program offered by companies where ethical hackers find and report security vulnerabilities in their applications, websites, or systems. In return, hackers receive monetary rewards based on the severity of the bug found.

How It Works:

  1. Company launches program ‚Üí Defines scope and rules
  2. Hacker finds vulnerability ‚Üí Tests within legal boundaries
  3. Submit report ‚Üí Detailed proof of concept
  4. Company validates ‚Üí Reviews and confirms the bug
  5. Receive reward ‚Üí Payment based on severity
‚ö†Ô∏è Legal Requirements:
‚Ä¢ ONLY test on programs with explicit permission
‚Ä¢ NEVER test on random websites without authorization
‚Ä¢ Follow the program's rules of engagement
‚Ä¢ Testing without permission is ILLEGAL and can result in criminal charges

3. Essential Web Vulnerabilities

A. Cross-Site Scripting (XSS)

What it is: Injecting malicious JavaScript into web pages viewed by other users
Impact: Steal cookies, hijack sessions, deface websites
Types:
  • Reflected XSS - Script in URL/request
  • Stored XSS - Script saved in database
  • DOM XSS - Client-side JavaScript vulnerability
Example Payload: 
<script>alert(document.cookie)</script>
<img src=x onerror=alert(1)>
<svg onload=alert('XSS')>
Where to look: Search boxes, comment sections, URL parameters, user profiles, file uploads

B. SQL Injection (SQLi)

What it is: Inserting malicious SQL code into database queries
Impact: Access entire database, bypass authentication, data theft
Types:
  • Classic SQLi - Direct query manipulation
  • Blind SQLi - No visible output, use true/false
  • Time-based - Using delays to extract data
Example Payloads: 
' OR '1'='1
admin' --
' UNION SELECT username, password FROM users--
1' AND SLEEP(5)--
Where to look: Login forms, search features, URL parameters, sorting/filtering options

C. Cross-Site Request Forgery (CSRF)

What it is: Forcing users to execute unwanted actions on a web application
Impact: Change passwords, transfer funds, modify data
Example Attack: 
<form action="https://bank.com/transfer" method="POST">
    <input type="hidden" name="amount" value="1000">
    <input type="hidden" name="to" value="attacker">
</form>
<script>document.forms[0].submit();</script>
Where to look: State-changing operations without tokens, important actions via GET requests

D. Insecure Direct Object Reference (IDOR)

What it is: Accessing objects by changing ID values without authorization
Impact: Access other users' data, modify/delete unauthorized content
Example: 
https://site.com/profile?id=123 ‚Üí Change to id=124
https://site.com/invoice/INV-2021-001 ‚Üí Try INV-2021-002
/api/users/100/details ‚Üí Try other user IDs
Where to look: URLs with IDs, API endpoints, file downloads, user profiles

E. Server-Side Request Forgery (SSRF)

What it is: Making the server perform requests to unintended locations
Impact: Access internal systems, port scanning, read local files
Example Payloads: 
url=http://localhost/admin
url=http://169.254.169.254/ (AWS metadata)
url=file:///etc/passwd
Where to look: URL input fields, webhooks, file imports, PDF generators

F. XML External Entity (XXE)

What it is: Exploiting XML processors to read files or perform requests
Impact: Read local files, internal network access, DoS
Example Payload: 
<?xml version="1.0"?>
<!DOCTYPE root [<!ENTITY xxe SYSTEM "file:///etc/passwd">]>
<root>&xxe;</root>
Where to look: XML uploads, SOAP endpoints, SVG uploads, SAML authentication

5. Testing Methodology

Step-by-Step Approach:

  1. Reconnaissance (Information Gathering)
    • Find subdomains using tools
    • Identify technologies with Wappalyzer
    • Check robots.txt, sitemap.xml
    • Search for exposed files (.git, .env, backup files)
  2. Mapping the Application
    • Browse entire application as normal user
    • Create accounts with different roles
    • Map all functionality and endpoints
    • Identify input points and parameters
  3. Vulnerability Testing
    • Test authentication (password reset, registration)
    • Check authorization (IDOR, privilege escalation)
    • Input validation (XSS, SQLi)
    • Business logic flaws
    • File upload restrictions
  4. Exploitation & Validation
    • Confirm vulnerability works
    • Document steps to reproduce
    • Assess real impact
    • Take screenshots/videos

7. Learning Path for Beginners

Week 1-2: Fundamentals

  • Learn HTTP/HTTPS basics, request methods, headers
  • Understand HTML, JavaScript, basic web technologies
  • Study OWASP Top 10 vulnerabilities
  • Set up Burp Suite/OWASP ZAP

Week 3-4: Hands-on Practice

  • Complete PortSwigger Web Security Academy (Free)
  • Practice on DVWA (Damn Vulnerable Web App)
  • Try OWASP WebGoat challenges
  • Join TryHackMe or HackTheBox for guided learning

Week 5-6: Real Targets

  • Create accounts on HackerOne and Bugcrowd
  • Read disclosed reports to understand patterns
  • Start with programs accepting beginners
  • Focus on wide-scope programs (*.target.com)

Week 7+: Specialize

  • Pick 2-3 vulnerability types to master
  • Develop custom tools and scripts
  • Build reconnaissance automation
  • Contribute to bug bounty community
